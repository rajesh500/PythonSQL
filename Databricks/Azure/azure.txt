
Azure storage redundancy:
LRS (Locally redundant storage):
with in the datacenter 3 copies will be maintained.
Low cost
No protection against datacenter failure

ZRS (Zone redudant storage):
Data will store in 3 zones with in the region.
Zone1, Zone2, Zone3.
For any region zone1 is failed, zone2 and 3 will become available.

All zones are failed with disaster region.
Entire datacenter based on the zone will not be available.


GRS (Geo-Redundant Storage):
data store in primary and secondary regions.


RA-GRS (Read Access Geo-Redundant Storage):
Same as GRS, but:

Primary Region
      ↓
Secondary Region
      ↓
Read Access Available
You can read from the secondary region even if the primary is healthy.



GZRS (Geo-Zone Redundant Storage)
Best protection.

Primary Region
   ├─ Zone 1
   ├─ Zone 2
   └─ Zone 3
          ↓
Secondary Region

Combines:
ZRS (zone protection)
GRS (region protection)

CDC vs SCD:
I don't need CDC to implement SCD2. SCD2 and CDC solve different problems. SCD2 maintains historical records in the target. CDC identifies changed records in the source. If the source volume is manageable, I can detect changes using hash comparison and implement SCD2 without CDC. CDC becomes valuable when source tables are very large because it avoids scanning and comparing the entire dataset and provides only the changed rows.

DMS:
In the context of Microsoft Azure, DMA stands for Data Migration Assistant. It is a free tool used to evaluate on-premises SQL Server databases and determine their readiness for migration to Azure. 

What Does DMA Do?
Before you move your databases to the cloud, DMA performs several vital checks:
Compatibility Analysis: It identifies any blockers, deprecated features, or unsupported code that might prevent a smooth migration to target environments like Azure SQL Database or Azure VMs.
Feature Recommendations: It suggests improvements for performance and reliability in your new target environment.
Schema & Data Transfer: It assists in moving your database schema, data, and uncontained objects (like logins and agent jobs) to the cloud

When to Use What?
Use Databricks Jobs when:
You are running Spark notebooks
ETL is fully inside Databricks
You want simple scheduling

Use ADF when:
You need data movement (copy activity)
You integrate multiple systems (SQL, APIs, Storage)
You want low-code pipelines

Use Airflow when:
You need complex DAG logic
You have multi-cloud systems
You need full programmatic control
