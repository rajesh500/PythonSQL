Metadata:
data about data, that describes the structure, origin and usage of datasets, acting as a label or guidebook.

Technical Metadata: Defines the schema, data types, column names, table structures and storage formats.
Operational Metadata: Tracks data pipeline performance, job schedules, runtime, metrics and lineage(where data came from and how it changed).
Business Metadata: Adds context such as data ownership, definitions.


Data Lineage:
Life cycle of data
how data moves and changes from its original source to where it’s ultimately used (reports, dashboards, ML models, downstream apps). 
Think of it as a traceable “chain of custody” for data.

What it captures:
Origins (sources): databases, files, APIs, SaaS systems
Movement (pipelines): ingestion, ETL/ELT jobs, streaming flows
Transformations: joins, filters, aggregations, business rules, calculations
Destinations (consumers): data warehouse/lakehouse tables, BI metrics, regulatory reports
Metadata: timestamps, owners, versions, schema changes, job runs

Z-ordering:
let say you have multiple partitions and you have country code in those partitions.
partition1  IN, USA, AUS
partition2  AUS, IN, IN 
partition3  USA, IN, USA
if you try to filter out the data on country bases then all the partitions need to be scanned and filter it out.

If we use zordering then in the back it will re-partition the on country bases, it will go that particular partition and fetch the data.
partition1  USA, USA, USA
partition2  IN, IN, IN 
partition3  AUS, AUS, AUS.

zordering will do the re-ordering and store in the metadata not in the original location.

## Delta lake:
Delta lake was built on top of data lake storage.
delta lake is an open storage framework that brings reliability and performance to data lakes.
key features
1. schema enforcement rejects writes to a table if the incoming data structure does not match the existing table schema.
2. time travel / versioning – history
3. update
4. delete
5. merge scd-1 
6. ACID transactions
7. schema evolution : automatically adjust its structure.
8. performance optimizations such as OPTIMIZE, Z-ordering..
In my databricks projects, i use delta lake for incremental processing, CDC implementations, and maintaining reliable Bronze, Silver and 
Gold layers in a medallion architecture.

OOM:
Scenario-1: Large Partition

Suppose:
1 TB File processing required 100 Partitions

Each partition:
1 TB / 100 = 10 GB

Executor memory: 8 GB

When Spark tries to process a 10 GB partition:
Need 10 GB
Available 8 GB

Result:
Executor OOM


Scenario 2: Data Skew

Example:
Customer A = 90% of records
Other customers = 10%

During join/groupBy:

Partition 1 = 50 GB
Partition 2 = 100 MB
Partition 3 = 200 MB

Executor handling:
50 GB partition

runs out of memory.

Result:
Executor OOM



Scenario 3: Large Shuffle

Operations like:
groupBy()
join()
distinct()
orderBy()   cause shuffle.

Example:
2 TB data
     ↓
groupBy(customer_id)

Spark must:
Move data between executors
Sort data
Aggregate data

Intermediate data may become larger than available memory.

Result:
Spill to disk
Then OOM


Scenario 5: Cache/Persist

Example:
df.cache()

Data:
100 GB

Cluster memory:
40 GB

Spark tries to keep data in memory.
Memory fills up.

Result:
Executor OOM


## Shuffle:
What is Shuffling in Spark?

Shuffling is the process of moving data across executors/partitions so that Spark can perform operations such as joins, aggregations, sorting, and distinct.

It is one of the most expensive operations in Spark because it involves:

Network I/O
Disk I/O
Serialization/Deserialization
Sorting


Simple Example

Suppose you have data:

Partition 1                Partition 2

(A,100)                    (A,200)
(B,300)                    (B,400)
(C,500)                    (C,600)

Now run:

df.groupBy("customer_id").sum("amount")

Spark needs all records for the same customer together.


After shuffle:
Partition 1
A,100
A,200

Partition 2
B,300
B,400

Partition 3
C,500
C,600

Spark moved data across executors. This movement is called shuffle.


Operations That Cause Shuffle:
df.groupBy("customer_id").sum()
df1.join(df2, "customer_id")
df.distinct()
df.dropDuplicates()
5. orderBy / sort
df.orderBy("amount")
df.repartition(100)



Real Interview Example

Suppose:
Fact Table = 1 TB, Dimension Table = 500 GB

Join:
fact.join(dim, "customer_id")

Spark may shuffle:
1.5 TB+
across executors.

Result:
Slow job
High memory usage
Possible OOM



How to Reduce Shuffle:
Broadcast Join
Filter Early
Select Required Columns
Proper Partitioning



### checkpoint and autoloader:
I have a large file 500GB or 2TB(file csv or parquet). pipeline is failed after processing 300GB data pipeline is failed how to restart it without processing entire file again.
Using check point concept we can process it but completely using spark we were not able to achieve it.
spark doesn't support the checkpoint, it has restart it from the begining.

but this can achieve using autoloader.
source path:
target path:
checkpoint locator path:

During the writing, mention this checkpoint locator path, this auto loader  will handle checkpoint incase of failure.
during this process as well split the large file into multiple files.


from pyspark.sql.functions import current_timestamp, input_file_name

source_path = "abfss://apexusdev@apexusdev.dfs.core.windows.net/input/parquet/"
target_path = "abfss://apexusdev@apexusdev.dfs.core.windows.net/delta/customer/"
checkpoint_path = "abfss://apexusdev@apexusdev.dfs.core.windows.net/checkpoints/customer/"

df = spark.readStream \
    .format("cloudFiles") \
    .option("cloudFiles.format", "parquet") \
    .load(source_path)

final_df = df.withColumn("source_file", input_file_name()) \
             .withColumn("processed_time", current_timestamp())

query = final_df.writeStream \
    .format("delta") \
    .option("checkpointLocation", checkpoint_path) \
    .option("mergeSchema", "true") \
    .outputMode("append") \
    .trigger(availableNow=True) \
    .start(target_path)

query.awaitTermination()
