Liquid Clustering:
Liquid Clustering is a modern data layout optimization feature in Delta Lake that replaces many traditional uses of partitioning and Z-Ordering. It allows Delta tables to continuously reorganize data based on clustering keys without requiring static partitions.

Why was Liquid Clustering introduced?

Traditional partitioning has limitations:

Too many partitions → millions of small files.
Too few partitions → poor query performance.
Changing partition columns later is difficult.
Frequent updates can lead to data skew.



Liquid Clustering + optimize using both this reorganize of data will be done.

Traditional Partitioning

      month=01/
      month=02/

Files are reorganized automatically

There are no static partition folders. Delta organizes the data internally to improve data skipping.

Liquid Clustering vs Z-Ordering:
Z-Ordering is a one-time optimization technique.

example:
CustomerID  OrderDate
101         2025-01-01
500         2025-01-03
101         2025-01-05
300         2025-01-07
101         2025-01-08

Run:

OPTIMIZE sales
ZORDER BY (customer_id);

Spark rewrites the files so rows with similar customer_id values are placed closer together.

Result:

File1
101
101
101

File2
300
500


Limitation

Tomorrow another 100 GB arrives.

The new files are not automatically Z-Ordered.


Every new data got loaded into the table z-ordering need to be run to reorganize the data.

Liquid Clustering is a persistent table property that defines how data should be organized over time.
Continuously organize data.
No need to rerun it to reorganize the data.

delta lake is an open storage framework that brings reliability and performance to data lakes.

3. update

## Shuffle:
What is Shuffling in Spark?

Shuffling is the process of moving data across executors/partitions so that Spark can perform operations such as joins, aggregations, sorting, and distinct.

It is one of the most expensive operations in Spark because it involves:

Network I/O
Disk I/O
Serialization/Deserialization
Sorting


Simple Example

Suppose you have data:

Partition 1                Partition 2

(A,100)                    (A,200)
(B,300)                    (B,400)
(C,500)                    (C,600)

Now run:

df.groupBy("customer_id").sum("amount")

Spark needs all records for the same customer together.


After shuffle:
Partition 1
A,100
A,200

Partition 2
B,300
B,400

Partition 3
C,500
C,600

Spark moved data across executors. This movement is called shuffle.


Operations That Cause Shuffle:
df.groupBy("customer_id").sum()
df1.join(df2, "customer_id")
df.distinct()
df.dropDuplicates()
5. orderBy / sort
df.orderBy("amount")
df.repartition(100)



Real Interview Example

Suppose:
Fact Table = 1 TB, Dimension Table = 500 GB

Join:
fact.join(dim, "customer_id")

Spark may shuffle:
1.5 TB+
across executors.

Result:
Slow job
High memory usage
Possible OOM



How to Reduce Shuffle:
Broadcast Join
Filter Early
Select Required Columns
Proper Partitioning



### checkpoint and autoloader:
I have a large file 500GB or 2TB(file csv or parquet). pipeline is failed after processing 300GB data pipeline is failed how to restart it without processing entire file again.
Using check point concept we can process it but completely using spark we were not able to achieve it.
spark doesn't support the checkpoint, it has restart it from the begining.

but this can achieve using autoloader.
source path:
target path:
checkpoint locator path:

During the writing, mention this checkpoint locator path, this auto loader  will handle checkpoint incase of failure.
during this process as well split the large file into multiple files.


from pyspark.sql.functions import current_timestamp, input_file_name

source_path = "abfss://apexusdev@apexusdev.dfs.core.windows.net/input/parquet/"
target_path = "abfss://apexusdev@apexusdev.dfs.core.windows.net/delta/customer/"
checkpoint_path = "abfss://apexusdev@apexusdev.dfs.core.windows.net/checkpoints/customer/"

df = spark.readStream \
    .format("cloudFiles") \
    .option("cloudFiles.format", "parquet") \
    .load(source_path)

final_df = df.withColumn("source_file", input_file_name()) \
             .withColumn("processed_time", current_timestamp())

query = final_df.writeStream \
    .format("delta") \
    .option("checkpointLocation", checkpoint_path) \
    .option("mergeSchema", "true") \
    .outputMode("append") \
    .trigger(availableNow=True) \
    .start(target_path)

query.awaitTermination()

CDF: (Change Data Feed)
CDF (Change Data Feed) in Databricks Delta Lake is a feature that records row-level changes (inserts, updates, and deletes) made to a Delta table. 
Instead of reading the entire table every time, downstream processes can read only the changes since a specific version or timestamp.

This is a property in the delta table, need to be enable it during creating a table or after creating the table.
CREATE TABLE customers (
    customer_id INT,
    name STRING,
    city STRING
)
USING DELTA
TBLPROPERTIES (
    delta.enableChangeDataFeed = true
);


or Alter table customer set TBLPROPERTIES (
    delta.enableChangeDataFeed = true
);


Ex: customer is table:

Now query this table after version1 what all the changes happended on version2 this can be tracked using some specific metadata columns.

## Lakeflow Declarative pipelines:
Lakeflow Declarative Pipelines (formerly Delta Live Tables - DLT) is Databricks' declarative framework for building batch and streaming ETL pipelines in SQL and Python. Instead of writing orchestration logic yourself, you declare the desired tables and transformations, and Lakeflow manages dependency resolution, execution, retries, autoscaling, monitoring, and incremental processing.


Traditional Spark ETL
Read Bronze
      ↓
Transform
      ↓
Write Silver
      ↓
Read Silver
      ↓
Transform
      ↓
Write Gold


Lakeflow
You simply define:
Bronze
    ↓
Silver
    ↓
Gold

Example:
Bronze load:
CREATE OR REFRESH STREAMING TABLE bronze_orders
AS
SELECT *
FROM STREAM read_files(
  'abfss://raw@storageaccount.dfs.core.windows.net/orders/',
  format => 'csv',
  header => true
);


Bronze to silver load:
CREATE OR REFRESH STREAMING TABLE silver_orders
(
  CONSTRAINT valid_order_id EXPECT (order_id IS NOT NULL) ON VIOLATION DROP ROW,
  CONSTRAINT valid_amount EXPECT (amount > 0) ON VIOLATION DROP ROW
)
AS
SELECT
  CAST(order_id AS BIGINT) AS order_id,
  CAST(customer_id AS BIGINT) AS customer_id,
  CAST(product_id AS BIGINT) AS product_id,
  CAST(amount AS DOUBLE) AS amount,
  CAST(order_date AS DATE) AS order_date
FROM STREAM bronze_orders;



Gold materialized view    (don't create a table if we have to do aggregation create a materalized view)
Aggregated reporting layer.

CREATE OR REFRESH MATERIALIZED VIEW gold_daily_sales
AS
SELECT
  order_date,
  COUNT(*) AS total_orders,
  SUM(amount) AS total_sales

Delta table checkpoint: 
There are two different types of checkpoints in Delta Lake, and they are often confused in interviews.

Delta Log Checkpoints (automatic, for transaction log performance)
Structured Streaming Checkpoints (for stream recovery)

Since you asked about Delta Lake table with checkpoint implementation, I'll explain Delta Log Checkpoints.


What is a Delta Log Checkpoint?

Every transaction on a Delta table creates a JSON log file in _delta_log.

Example:

sales/
│
├── part-00001.parquet
├── part-00002.parquet
│
└── _delta_log/
      00000000000000000000.json
      00000000000000000001.json
      00000000000000000002.json
      ...

Suppose your ETL runs every 5 minutes.

One day later:
288 commits/day
After a month:
8640 JSON files
If a new Spark cluster wants to read the table, it would have to read thousands of JSON transaction logs.
That would be slow.
Solution: Checkpoint

Delta automatically creates a checkpoint after a configurable number of commits (by default, every 10 commits).

Example:
_delta_log/
00000000000000000000.json
00000000000000000001.json
...
00000000000000000009.json

00000000000000000010.checkpoint.parquet

Instead of replaying all 10 JSON files, Spark reads:
Checkpoint
      +
Remaining JSON files

This makes table loading much faster.

Example
Suppose your table receives these operations:

Version 0
INSERT

Version 1
INSERT

Version 2
MERGE

Version 3
UPDATE

...

Version 10
DELETE

Delta creates:
00000000000000000010.checkpoint.parquet

Now when Spark opens the table:

Instead of:

Read

0.json
1.json
2.json
...
10.json

It reads:
10.checkpoint.parquet

+

11.json
12.json

Much faster.

Can we create checkpoint manually?

Normally No.

Delta creates checkpoints automatically.

You can change the checkpoint interval.

Example:

ALTER TABLE sales
SET TBLPROPERTIES (
  'delta.checkpointInterval' = '20'
);

Now Delta creates a checkpoint every 20 commits instead of every 10.

Why is it needed?
Imagine a production table.

↓
300 commits/day
↓
9000 commits/month

Without checkpoint:
Read
9000 JSON files
With checkpoint:
Checkpoint
+
Last few JSON files

Startup time becomes much faster.

Delta Checkpoint vs Streaming Checkpoint
Delta Log Checkpoint    Structured Streaming Checkpoint
Improves table read performance Tracks streaming progress
Stored in _delta_log    Stored in checkpoint directory
Created automatically   Specified by developer
Metadata optimization   Failure recovery and exactly-once processing
Interview Answer

A Delta Log checkpoint is a Parquet snapshot of a Delta table's transaction log. Every write operation creates a JSON log in the _delta_log directory, and replaying thousands of JSON files would slow table initialization. To avoid this, Delta Lake automatically creates checkpoint Parquet files at regular commit intervals (default every 10 commits). When a table is read, Spark loads the latest checkpoint and only the newer JSON logs, significantly reducing metadata read time and improving query performance.

Real-time Example

Suppose your ADF pipeline writes to a Delta table every 5 minutes:

ADF
   ↓
Databricks Notebook
   ↓
MERGE INTO Delta Table
   ↓
_delta_log grows
   ↓
After every 10 commits
Delta creates checkpoint.parquet automatically

In this scenario, you don't write any checkpoint code—Delta Lake manages transaction log checkpoints automatically in the background.
